# Symbol store — the canonical reference

A per-firmware mapping of `NAME → address` plus, eventually, message IDs +
RAM addresses + roles. Replaces "grep the codebase and memory tags" with
"look it up".

## Files

```
tools/symbols/
├── README.md                      this file
├── dump_nokix.py                  scrape NokiX LOCATE_*.rx → YAML
├── lookup.py                      addr/name lookup CLI (read all YAMLs)
├── 3310-v579.nokix.yaml           generated — DO NOT hand-edit
├── 3310-v639.nokix.yaml           generated
└── <model>-<ver>.overlay.yaml     hand-curated overlay (to be added)
```

Two sources, merged at lookup time:

1. **`*.nokix.yaml`** — generated by `dump_nokix.py` from the NokiX
   `LOCATE_*.rx` macros. ~180–250 named addresses per firmware,
   build-portable (re-run when a new firmware lands). NEVER edit.

2. **`*.overlay.yaml`** — hand-curated. The place to add:
   - functions NokiX has no locator for (`task_4_dispatcher`,
     `task_2_broker`, `sim_fatal_stager`, …);
   - the **message-ID table** (`MSG_KEY_DOWN=0x366`, `MSG_DIALOG_DRAW=0x57C`,
     `MSG_DISPLAY_MESSAGE=0x226`, …);
   - per-build RAM byte roles we've named ourselves (`reboot_reason`,
     `verdict`, `sim_gate`, `phone_mode`);
   - corrections / disambiguations where NokiX got a name wrong.

## Workflow

**Add a NEW firmware:**

```
./tools/symbols/dump_nokix.py firmware/Factory\ Reset\ 3310\ NR1\ v5.79.fls -j 16
# writes tools/symbols/3310-v579.nokix.yaml
```

Re-run any time the LOCATE_*.rx macros or the firmware changes.

**Look up by name:**

```
./tools/symbols/lookup.py 3310-v579 DIALOG_OPEN
# → 0x0028DB38 (nokix-locate)
```

**Look up by address (reverse lookup):**

```
./tools/symbols/lookup.py 3310-v579 0x002EEB94
# → DEV_RESET_WATCHDOGS (nokix-locate)
```

## Highlights from the first 3310 v5.79 dump

- 181 symbols resolved in 2.5 s; 178 of them also resolve on v6.39 (with
  different addresses — exactly what the per-firmware files capture).
- **Distinctions NokiX makes that we hadn't:**
  - `RESET_WATCHDOGS = 0x2C1996` (the high-level wrapper)
    vs `DEV_RESET_WATCHDOGS = 0x2EEB94` (the chip-level kicker —
    what we'd been calling `ccont_reset_wdt`).
- **Wall-breakers** (addresses we'd been hunting):
  - `DIALOG_OPEN = 0x28DB38` — the window-opener the session-7-9 wall
    has been about. Tail-calls `DIALOG_OPEN_MSG = 0x28D964` after
    constructing msg id `0xAF*8 = 0x578` (= MSG_DIALOG_DRAW family).
- **Adjacent-RAM clarifications:**
  - `NETMONITOR_STATUS = 0x11FD07`, `CURRENT_HANDLER = 0x11FD08`,
    `CURRENT_KEY = 0x11FD11`, `CALL_STATUS = 0x11FCFE`,
    `CURRENT_LANGUAGE = 0x11FCE8`, `CURRENT_PROFILE = 0x11FCDC`.
    A whole row of UI/state bytes around what we'd narrowly called the
    "SIM retry counter".

## How it works

NokiX's `LOCATE_*.rx` macros are ~500 entries of "this byte pattern with these
wildcards finds <semantic name>" — battle-tested across the entire DCT3
fleet. `tools/nokix/nokix.c` is a Linux-native Regina runner with the host
commands these macros call (`find`, `setbyte`, `create`, …). `dump_nokix.py`
spawns nokix once per name (`./nokix … locate_one.nrx NAME`), parses the
`NAME=DEC_ADDR` output, and writes the merged YAML.

`locate_one.nrx` is one-shot by design — NokiX's `fail` is `exit(1)` and
some locators legitimately call it when their pattern doesn't match the
loaded firmware. Per-process isolation makes misses non-fatal.

Filtered out automatically:

- `LOCATE_MATH.rx` (these are *importers* — they `create` math fns in free
  flash, returning the pool base; not symbol discovery).
- `LOCATE_OBSOLETE.rx` (legacy aliases that recurse through other locators).
- `PHONE_MODEL` / `FIRMWARE_TYPE` / `FIRMWARE_VERSION` / `FIRMWARE_DATE` /
  `FIRMWARE_ID` (return text, not an address).
- Any locate() that returns the auto-detected free-flash pool base
  (`0x00393EC0` on 3310 v5.79) — those are `create`-allocated stubs, not
  real firmware symbols.

## Status

**Landed:**

- **Overlay schema** — `<build>.overlay.yaml` carries addresses (with `role`),
  RAM addresses (`ram:` section), and message IDs (`messages:` section).
  Schema documented inline at the top of `3310-v579.overlay.yaml`. lookup.py
  reads all three.
- **Validator** — `symbols_check.py` opens each YAML's named `.fls`, verifies
  the FNV-1a id matches, every addr lies in a sane region, and code-region
  addresses don't land in blank (0xFF…) flash. Catches the typical drift
  modes (stale entries, pool-stub leak from LOCATE_MATH).
- **Code-gen → mad2** — `gen_c_table.py` emits
  `src/models/3310/symbols_3310_v579.h` (185 sorted (addr, name) entries +
  a binary-search `_nearest()` helper). mad2's post-mortem renderer uses it
  to label both the caller-LR and every asserting LR inline:
  `[SWDSP_STAGER+0xAA]`, `[BROKER_FAULT_NOTIFIER+0x12]`, etc.
- **Version-aware nokix** — `tools/nokix/nokix.c` parses the canonical MCU
  version header at flash offset `0x1FC` and sets
  `sys/firmware_{type,version,date}` + auto-detects `sys/phone_model` from
  the type code (NHM-5→3310, NSM-2→8850, …). `-m` becomes a manual override.

## To-do

- **`gen_c_table.py` per-build header per model**. Right now mad2.c
  hardcodes `#include "models/3310/symbols_3310_v579.h"`; should pick up
  the right header based on the loaded firmware (or merge all builds into
  one table tagged by fw-id).
- **Auto-detect free pool in `dump_nokix.py`**. Currently `DEFAULT_FREE_POOL`
  is 3310-specific (`0x393EC0`); the 8850 has its at `0x3D00xx`, so the
  8850 dump still includes ~41 pool-stub entries the validator flags.
  Easiest fix: have nokix emit the auto-detected pool on stderr (it
  already does with `-v`), parse it in the wrapper.
- **`run_at_end_locator`** — some NokiX locators that fail standalone may
  succeed when run after others (cache dependencies). Worth re-trying
  failed names after a full pass.
- **Cross-build sig roll-up** — a symbol present in both `3310-v579` and
  `3310-v639` with a clean address-delta is a good candidate for promotion
  to `src/models/3310/profile.c` as a sig (currently we only have ~12).
